# Conductor Workflow

## Overview

This document defines the standard development workflow for Conductor-managed projects. It ensures consistent quality, traceability, and maintainability across all tracks.

## Test Coverage

- **Minimum required coverage:** >80% (line coverage)
- Run tests before every commit
- Coverage reports generated by `cargo tarpaulin` (Rust) / `vitest --coverage` (TypeScript)

## Commit Strategy

- **Frequency:** Commit after every completed task
- **Format:** Conventional Commits (`feat:`, `fix:`, chore:`, `refactor:`, `test:`, `docs:`)
- **Scope:** Include the task ID in the commit footer (e.g., `Ref: #<track_id>-<task_number>`)
- **Summaries:** Recorded in commit message body

## Task Structure

- Each task in `plan.md` follows: `- [ ] Task: <description>`
- Sub-tasks are indented: `    - [ ] <action>`
- Status tracking: `[ ]` = pending, `[x]` = completed

## Phase Completion Verification and Checkpointing Protocol

At the end of each phase, a mandatory verification step must be executed:

1. **Run all tests:** Execute the full test suite and confirm zero failures
2. **Run linter:** Execute the linter (`cargo clippy` / `eslint`) and confirm zero warnings
3. **Run type checker:** Execute the type checker and confirm zero errors
4. **Verify coverage:** Confirm >80% line coverage threshold is met
5. **Manual review:** Review the diff for unintended changes and confirm alignment with `spec.md`
6. **Checkpoint:** If all checks pass, mark the phase as complete in `plan.md` and proceed

## Branch Strategy

- `main` — production-ready, protected
- `feat/<track_id>-<short-name>` — feature branches
- `fix/<track_id>-<short-name>` — bug fix branches
- No direct pushes to `main`; use PRs with review
